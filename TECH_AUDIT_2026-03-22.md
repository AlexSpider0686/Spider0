# Технический аудит чата и текущего билда (Vite + React + Vercel)

Дата аудита: 2026-03-22

## 1) Подтверждённый статус проекта

- Проект представляет собой одностраничное приложение на Vite + React с точкой входа `index.html -> src/main.jsx -> src/App.jsx`.
- Текущая структура действительно соответствует заявленной модели (`src/App.jsx`, `src/main.jsx`, `src/index.css`).
- Ошибка с иконкой `Layers3` устранена: в актуальном коде используется `Layers` из `lucide-react`.
- Конфигурация зависимостей соответствует MVP-стеку без Tailwind и без сложного роутинга/бэкенда.

## 2) Что уже реализовано относительно ТЗ из чата

### 2.1. Системы

В коде зафиксированы ровно 6 систем:

- СОТ (`sot`)
- СОТС (`sots`)
- СКУД (`skud`)
- ССОИ (`ssoi`)
- АПС (`aps`)
- СОУЭ (`soue`)

Каждая система участвует в расчёте по своим базовым нормативам (`BASE_RATES`) и отображается в UI.

### 2.2. Блок бюджета/коэффициентов

Реализован редактируемый набор параметров:

- `cableCoef`
- `equipmentCoef`
- `laborCoef`
- `complexityCoef`
- `overheadPercent`
- `ppePercent`
- `payrollTaxesPercent`
- `profitabilityPercent`
- `vatPercent`
- `taxMode` (ОСНО/УСН)

Это покрывает ключевую часть коммерческого расчёта (прямые + накладные + прибыль + НДС).

### 2.3. Вендоры

Реализована базовая модель:

- список вендоров по типу системы (`VENDORS`)
- коэффициенты индекса по вендорам (`VENDOR_INDEX`)
- выбор вендора для каждой системы
- ручная надбавка `customVendorIndex`

### 2.4. Экспорт

Есть экспорт CSV (`downloadCsv`, `exportEstimate`) с выгрузкой сводных данных по системам и общим итогом.

### 2.5. UI-структура

Реализован пошаговый сценарий:

1. Объект
2. Системы
3. Характеристики бюджета

Плюс финальный сводный блок итогов и таблица по системам.

## 3) Что потенциально не добито до полного продуктового уровня

### 3.1. «Справка по коэффициентам»

Поля коэффициентов есть, но нет развитого справочного слоя (диапазоны применения, рекомендации, методология выбора). Для пресейла это допустимо, для масштабирования — желательно добавить.

### 3.2. Глубина vendor-блока

Сейчас реализован укрупнённый коэффициентный подход по вендору. Не реализована корзина по позициям (BOM-level), нет интерактивного состава материалов по вендору.

### 3.3. Расчётная детализация

Расчёт в `calculateSystem` — инженерно-укрупнённый (метры кабеля, условные units, базовые ставки). Для «защищаемой сметы» в тендерах обычно нужна позиционная детализация:

- отдельные позиции оборудования
- нормы монтажа/ПНР по позициям
- спецификация материалов
- трассы/лотки/крепёж в разрезе типов

### 3.4. Архитектурный техдолг

Слишком много ответственности в `src/App.jsx`:

- константы
- бизнес-логика
- форматирование
- экспорт
- UI и обработчики состояния

Следующий безопасный шаг — вынос расчётов и констант в модули `src/lib/*` + `src/config/*`.

## 4) Риски для Vercel и сопровождения

- **Низкий риск**: повторного бага `Layers3`, если не откатывать импорт и держать ревью на импорты иконок.
- **Средний риск**: регрессия при рефакторинге `App.jsx` без поэтапного выноса и smoke-check `npm run build`.
- **Средний риск**: бизнес-ошибки в формулах из-за отсутствия unit-тестов на расчётную логику.
- **Средний риск**: CSV-экспорт без расширенной спецификации форматов (например, BOM, листы/секции, локализация разделителей под Excel-профили).

## 5) Рекомендуемый план следующего релиза (без «переписать всё»)

### Фаза 1 — Стабилизация ядра (обязательно)

1. Вынести `calculateSystem`, форматтеры и агрегаторы итогов в `src/lib/estimate.js`.
2. Вынести константы (`SYSTEM_TYPES`, `BASE_RATES`, `VENDORS`, `VENDOR_INDEX`) в `src/config/estimateConfig.js`.
3. Добавить минимальные unit-тесты на расчёты (включая налоговый режим ОСНО/УСН).

### Фаза 2 — Улучшение UX для пресейла

4. Добавить «Справку по коэффициентам» (tooltip/accordion + рекомендуемые диапазоны).
5. Добавить в блок систем отображение «что изменил вендор» (прозрачный расчёт индекса).
6. Расширить CSV: отдельные секции по каждой системе и итоговые подписи.

### Фаза 3 — Путь к коммерческой смете

7. Ввести опциональный режим «детализация по позициям» (BOM-lite).
8. Добавить шаблоны норм по типам объектов/зон.
9. Подготовить API-контракт для дальнейшего хранения расчётов на бэкенде.

## 6) Вывод

Текущий билд уже рабочий и хорошо подходит для MVP-пресейла: структура корректная, критичная проблема с `lucide-react` устранена, 6 целевых систем присутствуют, бюджет/налоги/вендоры и CSV-экспорт реализованы.

До «полноценного сметного продукта» не хватает главным образом трёх вещей:

1. модульной архитектуры расчётного ядра,
2. справочной прозрачности коэффициентов,
3. позиционной детализации для защищаемых коммерческих предложений.

Именно эти направления стоит брать в следующий цикл задач в Codex.
